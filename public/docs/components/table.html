Table

Tables display related information in the form of rows and columns. They allow users to quickly compare, sort, and visualize large amounts of related data.

Usage

Use a Table to organize and display large amounts of structured information. Table is ideal for looking up or comparing individual values.

A table should be accompanied by a section heading.

Design

Column Header

Column Headers should be organized in a left-to-right sequence that reinforces clarity of information in the table, such as by level of importance or in alphanumeric order. Specifically, a column of identifiers should be the leftmost column (e.g. "Name" or "Id" in a table of patients).

Column Headers should be in Title Case and should be concise to avoid wrapping to a second line. Omit information that is obvious from the table's title or from the column's data. For example, a column of patient names should be called "Name" in a table called "Patients".

Data Column

Data Columns run vertically in a table. A column holds data that corresponds to its Column Header.

By default, text within tables are aligned to the left and numbers are aligned to the right. Right-aligned numbers help the reader make comparisons of magnitude when scanning down columns.

Data Row

Data Rows run horizontally and contain related data items where the value in each column corresponds to its Column Header.

Table Aesthetics

  • Do size columns according to the data they contain. If all the columns are an even width, data becomes lost and detached from its neighbors. Some table cells will be too narrow and cramp their data uncomfortably. Let the columns fit the content rather than setting exact widths for each. This is the default behavior for Table.
  • Don't display too many rows or columns at once. Beyond potential performance issues, dense tables become less usable the larger they are. Tables are meant to be readable, scannable, and easily comparable. If a user loses track of information, the table is not helping to achieve their goal. Consider paginating for a large number of records or removing noncrucial columns.
  • Don't stretch a table full-width across the screen. While it might look preferable from a distance, it will be harder to read as the data will be unnecessarily separated. By default, the maximum width of Table is a sum of the widths of the columns.

Implementation

The Table component is based on Reactabular's core reactabular-table package. Forge leverages third-party libraries when possible, but must still achieve Forge design standards and present a unified API. Only the functionality described in the Forge documentation is officially supported for this component even if an underlying third-party API currently remains usable.

Templates

Columns can utilize a function to transform data for presentation purposes. For example, the function could turn entries into a link, add a currency symbol, or assign specific formatting.

Supply a function returning a string or a React element to the template property in the data column definition object (see the demo for an example). This is a thin wrapper around Reactabular's formatter.

Table Props

Prop Type Default Description
columns (required) arrayOf - shape:
  • displayName: string (required)
  • key: string (required)
  • numeric: boolean
  • template: func
 Defines the columns. Must include displayName and unique key. Can also specify the column as numeric to right-align content or set a custom template (See https://reactabular.js.org/#/column-definition/formatters).
rows (required) arrayOf - object Defines the rows as objects. Each key should match a column's key to display the value in that column.
className string Adds a class to the root element of the component
rowKey string 'id' Identifies which property to use as each row's key.
showHover bool true Indicates whether to show hover states for improved reading across a single row
...rest Passthrough props to the root component ReactabularTable.Provider that wraps the Table
